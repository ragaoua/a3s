Create a local Kind cluster:

```bash
KIND_EXPERIMENTAL_PROVIDER=podman kind create cluster --name a3s-local
```

Load the agent image (see [../agent/README.md](../agent/README.md) ) into Kind:

```bash
KIND_EXPERIMENTAL_PROVIDER=podman kind load image-archive --name a3s-local <(podman save localhost/agent --format oci-archive)
# NOTE: `KIND_EXPERIMENTAL_PROVIDER=podman kind load docker-image agent --name a3s-local`
# doesn't work. It seems to be a known issue with loading podman images in king, for which
# the above is a workaround (see https://github.com/kubernetes-sigs/kind/issues/2038).
```

Test that running the agent container works:

```bash
kubectl run smoke \
    --image=localhost/agent \
    --image-pull-policy=Never \
    --env LISTEN_PORT=10000 \
    --env LLM_API_URI=test \
    --env LLM_API_KEY=test \
    --env MODEL='qwen3' \
    --env AGENT_API_KEY=1234 \
    --env AGENT_NAME=Cody \
    --env AGENT_DESCRIPTION='A helpful coding assistant' \
    --env AGENT_INSTRUCTIONS="You are a coding agent."
# Check
kubectl logs smoke
# Delete
kubectl delete pod smoke
```

Create a service account, with the role and role bindings to create pods:

```bash
kubectl apply \
    -f sa.yaml \
    -f role.yaml \
    -f role_binding.yaml
```

Generate a token for the service account:

```bash
kubectl create token app-sa
```

For the app `.env` values:

- `K8S_CA_DATA` comes from the kubeconfig generated by Kind at cluster creation
  (`~/.kube/config`), in `clusters[].cluster.certificate-authority-data` for the
  Kind cluster context.

After a pod has been deployed, use port-forwarding to access it and test:

```bash
kubectl port-forward pod/<pod_name> <host_port>:<container_port>
```
